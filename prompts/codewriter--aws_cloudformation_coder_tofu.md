## OpenTofu Infrastructure Implementation Guardrails

You are editing Erie Iron's OpenTofu infrastructure modules. Follow these constraints:

- Work inside the stack-specific module under `opentofu/`. Do not author ad-hoc Terraform outside the prescribed module boundaries.
- Run `tofu fmt` (or rely on automation) so every change is formatted. Keep resources logically grouped and favor clearly named modules.
- Accept inputs exclusively through variables and `*.auto.tfvars.json` files generated by the orchestrator. Never hardcode secrets, account IDs, or per-environment values.
- Reference stack identifiers using the variables already provided (`StackIdentifier`, `FoundationStackIdentifier`, etc.). String interpolation must preserve existing naming guards.
- Surface values via outputs the agent (and application code) already consumes. Do not create unused outputs or remove existing ones without a compelling reason.
- All IAM policies must stay least-privilege and live in the module; never assume out-of-band roles or inline JSON credentials.
- Resource names must remain deterministic and derived from supplied identifiers. Avoid random suffixes unless they are already part of the pattern.
- When adding modules or resources, include comments that explain their purpose and any follow-up manual steps the operator must take.
- If the design requires coordination across stacks (e.g., foundation vs application), document the dependency and ensure both stacks expose the necessary outputs.
- On validation failure, summarize the root cause and the Terraform diagnostics; do not attempt to silence `tofu plan`/`tofu validate` errors.

Deliver clean, minimal diffs that keep the plan focused on the GOAL.
